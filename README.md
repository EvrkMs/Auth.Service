# Auth.Service

В этом каталоге собирается новый сервис авторизации для ComputerAdmin. Он должен оставаться opensource, но быть достаточно модульным, чтобы в будущем можно было подключать закрытые или платные расширения.

## Цели
- Безопасная авторизация персонала/менеджеров без публичной регистрации.
- Только PKCE (OAuth 2.1) клиенты и HTTPS-трафик, даже если сервис запущен вне Nginx.
- Полноценное управление сессиями поверх OpenIddict: хранение SessionId, выданных токенов, удалённое ревокирование.
- Поддержка Telegram как основного второго фактора (hash‑проверка через Telegram Widget, опциональный challenge-код, отправляемый в Telegram).
- Чёткая структура модулей, минимум хардкода доменных данных и максимум конфигурации через ENV.
- Возможность дальнейшей монетизации модулей (зарплата, аналитика, сейф и т.д.) через отдельные сервисы, подключённые по защищённому туннелю, без переписывания ядра.

## Структура
```
Auth.Service/
  README.md              ← этот файл
  TODO.md                ← рабочее ТЗ/бэклог
  src/
    Domain/
      Auth.Domain.Core     ← базовые сущности сотрудников/ролей/сессий
      Auth.Domain.Telegram ← модели/сервисы Telegram (binding, challenge)
      Auth.Domain.OpenId   ← сущности revocation/session tracking для OpenIddict
    Modules/
      Auth.OpenIddict      ← библиотека + API OpenIddict (login manager, issuance pipeline)
      Auth.Telegram        ← интеграция Telegram Widget / challenge flow
    Application/
      Auth.Application     ← use-case слой (команды/queries/handlers)
    Infrastructure/
      Auth.Infrastructure  ← реализации интерфейсов (EF/Redis/Telegram/OpenIddict integration)
    Auth.EntityFramework ← общий DbContext, который собирает конфигурации всех доменов + провайдеры (Postgres/MySQL/SQL Server)
    Auth.Host            ← компоновка, DI, безопасность host-а (CORS/HTTPS/Health/CSP), UI endpoints
    (доп. домены/модули) ← интеграции, настройки; внешние платные сервисы ставятся как отдельные приложения и подключаются через защищённые каналы
```

### Слои
- **Domain** — чистые модели/правила без инфраструктуры. Можно подключать новые домены (Telegram, OpenId, Settings) не затрагивая остальные.
- **Application** — use-case слой: команды, хэндлеры, сервисы координации (привязка Telegram, revoke сессий, выдача PKCE клиентов). Работает через интерфейсы, не знает про EF/Telegram/OpenIddict.
- **Infrastructure** — реализации интерфейсов (EF Core репозитории, Redis publisher, Telegram API клиент, OpenIddict stores, hosted services).
- **Modules** — вертикальные библиотеки, оборачивающие внешние технологии (OpenIddict, Telegram) и предоставляющие собственные API/контроллеры, чтобы не засорять `Auth.Host`.
- **Host** — минимальный ASP.NET проект, который включает все слои, настраивает DI, middleware, CORS/CSP/Health, и отдает UI/PKCE страницы.

### Основные модули
- **Auth.Domain.Core**: пользователи, роли, сессии, настройки (`AuthSettings`: `UseTelegram`, `UseTelegramChallenge`, `AuthDomain`, `AuthSubdomain` и т.д.).
- **Auth.Domain.Telegram**: TelegramBinding, challenge история, события привязки (можно подключать/отключать независимо).
- **Auth.Domain.Telegram** также хранит связи «один Telegram → много пользователей», чтобы поддержать выбор аккаунта при входе и массовое отвязывание.
- **Auth.Domain.OpenId**: сущности сессий, issued tokens, revocation события (используются OpenIddict-слоем).
- **Auth.EntityFramework**: единый DbContext, который подключает конфигурации всех доменов (через `IEntityTypeConfiguration`), миграции per provider (Postgres 18 базовый, опционально MySQL/SQL Server).
- **Auth.Application**: сценарии/handlers (привязка Telegram, управление сессиями, root bootstrap, выбор аккаунта при Telegram login), политики доступа; зависит только от Domain-интерфейсов.
- **Auth.Infrastructure**: реализации интерфейсов из Application (EF репозитории, Redis denylist publisher, Telegram API клиент, OpenIddict stores, hosted services). Цель — минимальная сцепка: Application видит только интерфейсы, EF/Redis/Telegram скрыты в Infrastructure.
- **Auth.OpenIddict**: собственная библиотека поверх OpenIddict — конфигурация серверов, контроллеры токен/интроспекции, login manager, PKCE/HTTPS enforcement, расширения issuance pipeline (сессии, revoke).
- **Auth.Telegram**: валидация Telegram Widget hash, хранение связки, challenge flow (генерация/отправка кода), API для привязки/отвязки.
- **Auth.Host**: стартовый проект, DI-композиция, middleware, UI маршруты `Account/Login|Logout|ChangePassword|TelegramBind`, `/sessions`, `/settings`, а также CORS/HTTPS/CSP/health-check конфигурация.

## UI и пользовательские сценарии
1. `Account/Login` — PKCE, HTTPS-only, поддержка смены пароля при неполной регистрации; если `UseTelegram = true`, отображает кнопку «Вход через Telegram», которая переводит на отдельную страницу Telegram-авторизации.
2. `Account/Logout` — закрытие текущей сессии + revoke связанных токенов.
3. `Account/ChangePassword` — обязательный шаг при первой авторизации (и после сброса root'ом).
4. `Account/TelegramBind` — привязка/отвязка Telegram; root может принудительно отвязать сотрудника.
5. `/sessions` — список активных/закрытых сессий, revoke по требованию, текущая определяется через access token.
6. `/account/telegram` — статус Telegram, настройки challenge-подтверждения.
7. `/settings` — защищённый root-интерфейс с собственным UX (хостится на том же Auth.Service), требует root-аутентификации через стандартный PKCE поток.
8. CRUD-интерфейсы (создание пользователей, сотрудников, ролей) остаются во внешнем `computeradmin-ui`; локальный UI в Auth.Service нужен только для PKCE-аутентификации и связанных шагов регистрации и реализуется полноценным UX-слоем (например, Razor Pages/Blazor) с валидациями и подсказками.
9. `Account/TelegramLogin` — отдельная страница для входа через Telegram Widget (вызвана кнопкой с `Account/Login`), содержит собственный POST-обработчик, чтобы обойти ограничения C# MVC на разделение обработчиков.
10. Telegram multi-account UX: если Telegram ID связан сразу с несколькими пользователями, страница `Account/TelegramLogin` предлагает выбрать целевой аккаунт; `Account/TelegramBind`/unbind должны отображать список всех аккаунтов, связанных с этим Telegram, и позволять отвязать неактивные записи.

## Сессии и безопасность
- Каждая авторизация создаёт запись `Session` → `IssuedTokens`.
- Сессия хранит отметку устройства/браузера, даты, статус (`Active`, `Closed`, `Revoked`).
- Root и сами пользователи могут закрывать сессии; при закрытии все связанные access/refresh токены сразу же отзываются через OpenIddict (чтобы он отдавал 401 и на интроспекции, и на middleware без кастомных списков).
- При отзыве сессии (через пользователя или root) сервис обязан найти все токены, выданные в рамках этой сессии, пометить их как revoked и предотвратить дальнейшее использование.
- Дополнительно каждое событие ревока публикуется в Redis-denylist (ключи `auth:denylist:token:{tokenId}` / `auth:denylist:session:{sessionId}`) с TTL равным сроку жизни соответствующего токена/сессии, чтобы внешние сервисы, проверяющие JWT только по JWKS, могли быстро отсеивать отозванные токены без интроспекции.
- Одноразовый пароль при сбросе root'ом генерируется сервисом, хранится в hashed-виде и требует немедленной смены.
- Любое действие root (сброс пароля или отвязка Telegram) автоматически закрывает и очищает все активные сессии/токены пользователя.
- Политика сроков жизни:
  - Access token — 15 минут.
  - Refresh token — 1 час с ротацией (каждый новый refresh инвалидирует предыдущий).
  - Session cookie — 1 день по умолчанию, 7 дней при `RememberMe = true`; при этом записи сессий ротуируются, а refresh выдаются заново.
  - Cookies используют `SameSite=None`, `Secure=true`, `Domain=AUTH_DOMAIN` (или `AUTH_PUBLIC_HOST`), чтобы поддерживать кросс-доменные PKCE клиенты.
- Флаг `IsRegistrationComplete` контролирует поток: смена пароля → проверка Telegram → (если включён `TelegramChallenge` и Telegram привязан) запрос кода → завершение регистрации.
- Детальный флоу после сброса пароля/первого входа:
  1. Обязательная смена пароля (если сброшен root'ом или пользователь впервые входит).
  2. Проверка `UseTelegram`.
     - `UseTelegram = false` → завершение регистрации сразу.
     - `UseTelegram = true`:
       - Нет активной привязки → запросить привязку Telegram; после успеха перейти к следующему шагу.
       - Telegram уже привязан → проверка `UseTelegramChallenge`.
         - `UseTelegramChallenge = false` → завершить регистрацию.
         - `UseTelegramChallenge = true` → сгенерировать и отправить код в Telegram; завершение только после успешного ввода.
  3. После успешного завершения всех шагов пользователь возвращается на исходный OIDC `return_url` (UI, который инициировал PKCE-поток).

## Конфигурация и ENV
- Обязательные переменные: `AUTH_DOMAIN`, `AUTH_SUBDOMAIN` (при необходимости), `AUTH_PUBLIC_HOST` (если нужно задать полный хост); допускается указать и `AUTH_DOMAIN`, и `AUTH_PUBLIC_HOST` одновременно, чтобы отдельные модули могли использовать нужный формат. Дополнительно требуются `POSTGRES_CONNECTION`, `HTTPS_PORT`, `TELEGRAM_WIDGET_TOKEN`, флаги `USE_TELEGRAM`, `USE_TELEGRAM_CHALLENGE`.
- Redis: `REDIS_CONNECTION`, `REDIS_NAMESPACE` (используется для denylist и публикации ревоков; TTL задаётся автоматически по срокам жизни токенов/сессий).
- Сервис должен падать при старте, если HTTPS не активен или домен не указан.
- `/settings` доступен после авторизации root-пользователя (может находиться на отдельном порту, но не ограничивается локальным доступом). При старте сервиса выполняется проверка подключения к БД/Redis; если соединение не установлено, сервис падает с 500 (`not init connect DB`) до устранения проблемы.
- `USE_CLEANUP_TASK` (или аналог) включает регулярную очистку закрытых/отозванных сессий и истёкших токенов OpenIddict из БД.
- Значение домена всегда берётся из ENV (чтобы соответствовать окружению контейнера); остальные настройки из `/settings` сохраняются в БД и управляются через API.
- `/settings` хранит несекретные параметры (UseTelegram, UseTelegramChallenge, разрешённые PKCE клиенты и т.д.) в базе, чтобы их можно было менять без рестарта контейнера.
- В `/settings` доступно управление PKCE клиентами: как минимум один дефолтный клиент `Admin-Panel` с предзаданным `ClientId` и обязательными полями домена + redirect/post-logout URLs.
- Root-пользователь создаётся при запуске (seeder) с `IsRegistrationComplete = false`, поэтому `/settings` недоступен, пока root не завершит первичную регистрацию.

## Сервис-ту-Сервис коммуникация
- Помимо стандартной интроспекции OpenIddict планируется выделенный «туннель»/шлюз между Auth.Service и другими микросервисами (например, постоянный gRPC канал или mTLS reverse proxy), чтобы проверка токенов не требовала каждого раза устанавливать новое HTTPS соединение.
- Redis + JWKS остаются fallback-вариантом (публикация ревоков), но основная цель — обеспечить короткий и защищённый путь между доверенными сервисами.
- Платные или кастомные модули разворачиваются как отдельные сервисы и подключаются к этому туннелю вместо встраивания в код Auth.Service.

## Технологии
- .NET 9 + C# 13.
- Entity Framework Core 9 (основная цель — Postgres 18, но планируется поддержка MySQL и SQL Server через отдельные провайдеры).
- OpenIddict (без Duende).
- Telegram Widget API.
